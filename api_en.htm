Quick Start

Access preparation

If you want to use the API, please log in to the web terminal, complete the API key application and permissions configuration, and then develop and trade according to the document details.

You can click on here to create API key。

Each account can create up to five Api Keys, and each Api Key can be used to set up three types of access rights: Read, Trade and Withdraw.

The competencies are described below：

  • Read permission: Read permission is used to query the data interface, such as: order query, transaction query and so on.
  • Transaction permissions: transaction permissions are used to place orders, cancel orders, and transfer class interfaces.
  • Withdraw permission: Permission to withdraw coins is used to create and cancel orders for withdrawing coins.

Be sure to remember the following information after successful creation：

  • Access Key API Access Key

  • Secret Key The key used to encrypt the signature authentication（Visible only on application）

Risk warning ：These two keys are closely related to the security of your account, so do not disclose them to anyone at any time together. API Key leaks can result in the loss of your assets (even if you don't have access to the currency). Please remove this API Key as soon as possible for leaks.

Interface type

BKEX Global provides users with two interfaces, allowing you to choose the appropriate method for checking quotes, trading or withdrawing cash, depending on your usage scenario and preferences.

REST API

REST, which stands for Representational State Transfer, is a popular HTTP-based communication mechanism, where each URL represents a resource.

One-time operations such as transactions or asset withdrawals are recommended to developers using the REST API.

WebSocket API

WebSocket is a new protocol in HTML5. It implements full-duplex communication between client and server, with a simple handshake you can establish a client-server connection. The server can proactively push information to clients based on business rules.

It is recommended that developers use the WebSocket API to obtain information on market conditions and buying and selling depths.

interface authentication

Both types of interfaces include public and private interfaces.

The public interface can be used to obtain basic information and quotes data. The public interface can be used without authentication.

The private interface can be used for transaction management and account management. Each private request must be signed using your API Key for signature verification.

Signature authentication

Signature note

API requests can be tampered with during transmission over the internet. To ensure that requests are not altered, private interfaces other than the public ones (Base Information, Quote Data) must use your API Key. Encryption is done to verify that the parameter or parameter value has not changed during transmission.
Each API Key needs to have the appropriate permissions to access the appropriate interface, and each newly created API Key Permissions need to be assigned. Before using an interface, review the permission types for each interface and make sure your API Key has the appropriate permissions.

What is required for a legitimate request：

  • Method request address: i.e. access server address api.bkex.com, e.g. https://api.bkex.com/v2/u/order/create.
  • API Access Key (X_ACCESS_KEY): The Access Key in the API Key you are requesting.
  • Required and optional parameters: Each method has a set of required and optional parameters that define the API calls. These parameters and their meanings can be found in the description of each method.
  • Signature: The value calculated cryptographically to ensure that the signature is valid and has not been tampered with.

Enumeration definition

A trading pair consists of the base currency and the quoted currency. Take the trading pair BTC_USDT for example, BTC is the base currency and USDT is the quoted currency.

Order and transaction related ID description

  • orderId : the unique number of the order
    • .
  • tradeId : The unique number of the transaction
    • .

Order Type

Direction of buying and selling:

  • BID: Buy
  • ASK: Sell

Order type:

  • LIMIT : Limit order, this type of order needs to specify the order price, order quantity.
  • MARKET : Market order, this type of order only needs to specify the amount or quantity of the order, do not need to specify the price , the order will be filled directly with the counterparty when it enters into a match until the amount or quantity falls below the minimum amount or quantity of the transaction For the stop.
  • STOP_LIMIT : A take profit stop order sets the order above or below the market price when the order reaches the The price is triggered before you are officially in the matchmaking queue.

Query order sorting rules:

  • TIME_ASC :Ascending by time.
  • TIME_DESC : Descending by time.
  • PRICE_ASC : ascending order by price.
  • PRICE_DESC : descending order by price.

Order Status

  • -1 : Stop-limit order not triggered, this status order is in the matchmaking queue.
  • 0 : Waiting to be filled, this status order is in the matchmaking queue.
  • 1 : Filled. This status order is not in the matchmaking queue and the full quantity of the order has been filled by the market.
  • 2 : Cancelled. This status order is not in the pinch order, this status order may have a partial quantity and was successfully cancelled.
  • 3 : Partially Filled, this status order is in the matchmaking queue and part of the order quantity has been filled by the market. Waiting for the rest of the transaction.

  • POST request, with all parameters sent in form-data format in the body of the request
